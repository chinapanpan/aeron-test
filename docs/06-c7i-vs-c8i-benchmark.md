# Aeron Cluster 基准测试对比报告：C7i.xlarge vs C8i.xlarge

## 测试环境

| 项目 | 详情 |
|------|------|
| **Region** | us-west-2 (Oregon) |
| **可用区** | us-west-2a |
| **网络拓扑** | Cluster Placement Group (同一物理机架组) |
| **Aeron 版本** | 1.48.0 |
| **JDK** | Amazon Corretto 17.0.18 |
| **OS** | Amazon Linux 2023 |
| **集群规模** | 3 节点（1 Leader + 2 Follower） |
| **测试客户端位置** | 集群 Follower 节点本地运行 |

| 指标 | C7i.xlarge | C8i.xlarge |
|------|-----------|-----------|
| **处理器** | Intel Xeon (Sapphire Rapids, 4th Gen) | Intel Xeon (Emerald Rapids, 5th Gen) |

| **网络带宽** | Up to 12.5 Gbps | Up to 12.5 Gbps |

## 测试方法

- **预热**：2,000 条消息，结果不计入统计
- **正式测试**：3 轮 × 10,000 条消息
- **消息模式**：同步请求-响应（发送 → 等待 ACK → 下一条）
- **消息大小**：~15 bytes（"hello-N" 格式）
- **延迟测量**：客户端 `System.nanoTime()` 差值，含网络 RTT + 集群处理 + Raft 共识

## 分析与结论

### 1. TPS：C8i 略优 (+4.3%)

C8i 在每一轮测试中都稳定高于 C7i 约 4-5%，说明 Emerald Rapids 处理器在 Aeron 集群的单线程共识路径上有微小但一致的优势。

### 2. 延迟：两者基本持平

- **平均延迟**：C8i 低约 4.2%，与 TPS 提升一致（更快处理 → 更低平均等待时间）
- **P50/P99/P999**：差异在 1% 以内，统计上无显著差异
- **尾部延迟** (P999)：两者都在 ~9.5ms 左右，说明瓶颈在 Raft 共识协议本身，而非 CPU

### 3. 为什么差异不大？

在本测试场景中，性能瓶颈不在 CPU 计算能力：

```
消息流转路径（约 6ms 总延迟）:
  Client → [UDP 网络] → Leader → [Raft 日志写入]
  → [Archive 持久化] → [Follower 复制确认]
  → [Quorum 达成] → [状态机处理] → [UDP 网络] → Client
```

- **Raft 共识** 需要至少一次网络往返（Leader → Follower → ACK），这是延迟主导因素
- **Archive I/O** 持久化写入引入固定开销
- **CPU 处理时间**（消息序列化、状态机逻辑）只占总延迟的很小比例

因此 CPU 代际升级（Sapphire Rapids → Emerald Rapids）对 Aeron Cluster 端到端性能的影响有限。

### 4. 建议

| 场景 | 推荐 |
|------|------|
| **成本敏感** | C7i.xlarge — 性能差异仅 4%，价格可能更低 |
| **追求最优性能** | C8i.xlarge — 稳定的 4% TPS 优势 |
| **大幅提升性能** | 应优化 Aeron 配置（BusySpin、CPU 绑核、huge pages）而非换实例类型 |
| **高吞吐需求** | 改用异步/批量发送模式，而非同步请求-响应 |
