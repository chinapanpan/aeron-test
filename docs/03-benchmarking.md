# Aeron Cluster 压测基准报告

## 1. 压测环境

| 项目 | 配置 |
|------|------|
| 集群节点 | 6 台 EC2 m7i.xlarge (4 vCPU, 16GB RAM) |
| 管理/客户端节点 | 172.31.16.209 (t3.large) |
| JDK | Amazon Corretto 17.0.18 |
| Aeron 版本 | 1.48.0 |
| 网络环境 | AWS VPC 内部网络 (同一子网, 同一可用区 us-west-2a) |

所有节点部署在同一子网、同一可用区内，网络延迟极低，排除了跨 AZ 或跨 Region 带来的额外延迟干扰。

## 2. 压测工具

使用 Aeron 内置的 `ClusterClient` 基准测试模式进行压测。

**启动命令：**

```bash
java ... -Daeron.cluster.benchmark=true -Daeron.cluster.message.count=10000
```

**工具特点：**

- 基于 Aeron 原生客户端，无额外中间层开销
- 支持同步和异步发送模式
- 自动统计端到端延迟和吞吐量

## 3. 基准测试结果

### 3.1 初次测试 (同步模式)

| 指标 | 数值 |
|------|------|
| 消息数量 | 10,000 |
| 总耗时 | 73,943 ms |
| 吞吐量 | 135 msg/sec |
| 平均往返延迟 | 7,288 us (7.3 ms) |

### 3.2 结果分析

**同步客户端限制：**

- 同步客户端在每发送一条消息后，必须等待集群响应才能发送下一条消息
- 吞吐量完全受限于 RTT（往返延迟）
- 单线程业务逻辑下：`Throughput = 1 / processing_time`

**延迟构成分析：**

```
客户端 -> Leader: ~0.2ms (同AZ网络延迟)
Leader -> Follower (Raft 复制): ~0.2ms
Follower 确认 -> Leader: ~0.2ms
Leader -> 客户端响应: ~0.2ms
---
理论网络延迟: ~0.8ms
实际测量延迟: ~7.3ms
额外开销: 日志持久化、序列化/反序列化、GC 停顿等
```

**AWS 网络延迟参考：**

- 同一可用区 (same AZ) 内通常 < 0.5ms
- 网络带宽不是瓶颈：m7i.xlarge 最高支持 12.5 Gbps

## 4. 性能模型

### 4.1 Little's Law (利特尔法则)

```
Throughput = Concurrency / Latency
```

| 参数 | 当前值 | 说明 |
|------|--------|------|
| Concurrency (并发度) | 1 | 同步模式，每次仅 1 条消息在途 |
| Latency (延迟) | 7.3 ms | 实测平均往返延迟 |
| Throughput (吞吐量) | ~135 msg/sec | 1 / 0.0073s |

### 4.2 集群提交延迟模型

```
Cluster Commit Latency = 1.5 * RTT
```

组成部分：
1. **Client -> Leader**: 客户端发送消息到 Leader 节点
2. **Leader -> Follower Quorum**: Leader 将日志复制到多数派 Follower
3. **Commit Back -> Client**: 提交确认返回客户端

### 4.3 理论最大吞吐量 (异步 Pipeline 模式)

当使用异步 Pipeline 模式时，多条消息可以同时在途（in-flight），不必等待前一条消息的响应：

- **理论最大吞吐量**: 可达 **20,000+ msg/sec**
- 瓶颈将从网络延迟转移到 CPU 处理能力、日志持久化速度和网络带宽

## 5. 压测场景设计

### 场景 1: 基础吞吐量测试 (已完成)

- **目标**: 建立基准性能数据
- **方法**: 同步发送 10,000 条消息
- **测量指标**: 端到端延迟、总吞吐量
- **状态**: 已完成，结果见第 3 节

### 场景 2: 不同消息大小

- **目标**: 观察消息大小对吞吐量和延迟的影响

| 消息大小 | 预期影响 |
|----------|----------|
| 100 bytes | 基准大小，最低延迟 |
| 1 KB | 轻微增加，序列化开销增大 |
| 10 KB | 显著增加，日志写入量增大 |
| 100 KB | 大幅增加，可能触发分片和 MTU 限制 |

- **关注点**: 吞吐量 (bytes/sec) 变化趋势、延迟分布（P50/P99/P999）

### 场景 3: 持续负载测试

- **目标**: 验证系统在持续负载下的稳定性
- **方法**: 持续发送消息 60 秒
- **关注点**:
  - 吞吐量是否随时间衰减
  - GC 停顿对延迟毛刺的影响
  - 内存使用是否稳定
  - 日志文件增长速度

### 场景 4: 故障恢复性能测试

- **目标**: 验证集群在故障下的恢复能力
- **方法**: 在压测过程中 kill Leader 节点
- **测量指标**:
  - Leader 选举时间 (election time)
  - 服务恢复时间 (recovery time)
  - 消息丢失数量（预期为 0，Raft 保证）
  - 新 Leader 稳定后的吞吐量恢复速度

## 6. 关键指标收集方法

### 6.1 Aeron 内部指标 (AeronStat)

```bash
# 查看所有 Aeron 统计计数器
java -cp aeron-all.jar io.aeron.samples.AeronStat
```

重点关注：
- **字节数**: 发送/接收的总字节数
- **心跳**: 心跳发送/接收计数
- **NAK (Negative Acknowledgement)**: 丢包重传请求数
- **Retransmit**: 实际重传次数

### 6.2 网络层指标

```bash
# UDP socket buffer 使用情况
ss -u -a -n

# 查看 socket buffer 溢出
cat /proc/net/udp
```

### 6.3 系统资源指标

```bash
# CPU 和内存使用情况
top -b -n 1

# 系统整体资源统计
vmstat 1

# IO 等待
iostat -x 1
```

### 6.4 集群时间戳

- **Leader 选举时间**: 从 Leader 失效到新 Leader 选举完成的时间差
- **日志复制延迟**: Leader 写入日志到 Follower 确认的时间差

## 7. 压测注意事项

### 7.1 JVM 预热 (Warmup Phase)

JIT 编译器需要足够的运行时间才能将热点代码编译为本机代码。

- **必须进行预热**: 在正式测量前，至少发送 1,000-5,000 条预热消息
- **预热期间数据不计入**: 避免冷启动数据影响基准结果
- **建议**: 使用 `-XX:+PrintCompilation` 监控 JIT 编译进度

### 7.2 GC 停顿影响

- GC 停顿会导致延迟毛刺（latency spikes）
- 建议使用低延迟 GC 收集器：
  - `-XX:+UseZGC`（推荐，Corretto 17 支持）
  - `-XX:+UseShenandoahGC`
- 使用 `-Xlog:gc*` 记录 GC 日志，分析停顿时间

### 7.3 操作系统调优 (已应用)

以下 sysctl 和系统配置已在所有节点上应用：

```bash
# 网络缓冲区调优
net.core.rmem_max = 8388608
net.core.wmem_max = 8388608

# 共享内存 (/dev/shm)
# Aeron 使用 /dev/shm 进行进程间通信
# 确保 /dev/shm 空间充足

# 文件描述符限制
ulimit -n 65536
```

### 7.4 网络带宽

- m7i.xlarge 实例网络带宽最高可达 **12.5 Gbps**
- 在当前测试场景下（小消息、低吞吐量），网络带宽**不是瓶颈**
- 当消息大小增大到 100KB 以上且并发度提高时，需要关注带宽利用率

### 7.5 其他注意事项

- **测试隔离**: 确保压测期间无其他应用抢占 CPU 和网络资源
- **多轮测试**: 每个场景至少运行 3 轮，取中位数作为最终结果
- **时钟同步**: 所有节点使用 NTP 同步时钟，减少跨节点时间戳误差
- **结果可复现**: 记录完整的 JVM 参数、系统配置和测试脚本，确保结果可复现
